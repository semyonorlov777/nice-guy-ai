# Эталонный чеклист аудита программы

**Версия:** 1.2
**Последнее обновление:** Март 2026
**Эталон:** nice-guy v2 (полная переработка по скиллу book-to-modes v2)

---

## A. Режимы и промпты

- [ ] **A1:** Количество режимов ≥ 7 (5 архетипов + подвиды по паттерну размножения). Не считая Свободный чат и Разговор с автором
- [ ] **A2:** Каждый режим имеет `system_prompt` в `program_modes` (не fallback на `programs.system_prompt`)
- [ ] **A3:** Каждый промпт содержит секции: РОЛЬ, КНИГА, ПРАВИЛА ОТВЕТА, ЛОГИКА ДИАЛОГА, АНТИПАТТЕРНЫ
- [ ] **A4:** Секция ПРАВИЛА ОТВЕТА включает: лимит 60-80 слов, один вопрос за раз, scaffolding fading (4→2→0)
- [ ] **A5:** Секция АНТИПАТТЕРНЫ перечислена явно: стены текста, похвала личности, overdiagnosis, несколько вопросов за раз
- [ ] **A6:** Промпт содержит конкретные концепции из книги (не общие фразы, а термины автора)
- [ ] **A7:** Промпт содержит suggested replies с типами (голос пользователя, сопротивление, навигация)

## B. Welcome-сообщения

> **Два пути хранения welcome:**
> - **Структурированный (Hub/NewChatScreen):** `welcome_ai_message` + `welcome_replies` (JSON array) + `welcome_title/subtitle/mode_label`
> - **Классический (ChatWindow):** `welcome_message` (текст с «кавычками» для replies)
>
> Проверяй ОБА пути. Для Hub-режимов приоритет — структурированный путь.

- [ ] **B1:** Каждый режим имеет welcome: либо `welcome_ai_message` (структурированный), либо `welcome_message` (классический). Для Hub-режимов — проверять `welcome_ai_message`
- [ ] **B2:** Welcome следует шаблону: ЭМОДЗИ + НАЗВАНИЕ → ХУК → КАК РАБОТАЕТ → ЧТО ПОЛУЧИТЕ → SUGGESTED REPLIES
- [ ] **B3:** ХУК продаёт результат, НЕ описывает процесс (не «мы будем...», а «ты получишь...»)
- [ ] **B4:** Suggested replies — от первого лица, голос пользователя (не вопросы к AI)
- [ ] **B5:** Есть безопасный exit-reply («Мне сложно сформулировать» или аналог)
- [ ] **B6:** Welcome не длиннее ~300 слов (не стена текста)

## C. Техническая реализация

- [ ] **C1:** Все режимы имеют `mode_template` с уникальным `key`
- [ ] **C2:** Для каждого `route_suffix` существует `page.tsx` в `app/program/[slug]/(app)/`
- [ ] **C3:** Иконка каждого режима существует в `mode-icons.tsx` → `iconMap`
- [ ] **C4:** `sort_order` корректен: нет дублей, логичный порядок (диагностика → анализ → практика → свободный)
- [ ] **C5:** `access_type`: хотя бы 1 free режим (точка входа для конверсии)
- [ ] **C6:** Нет `enabled: false` режимов, которые отображаются в UI как «coming soon» больше 1 месяца
- [ ] **C7:** `chat_type` совпадает с `mode_template.key` для уникальных режимов

## D. Кросс-режимные связки

- [ ] **D1:** Промпты упоминают использование данных из других режимов (`{{cross_mode_data}}` или аналог)
- [ ] **D2:** В файле MODE_DETAILS или ретроспективе описаны связки: ПЕРЕДАЁТ → / ПОЛУЧАЕТ ←
- [ ] **D3:** Режимы-аналитики (Анализ) передают данные в режимы-практики (Ролевая, Воркшоп)
- [ ] **D4:** Нет «сирот» — каждый режим (кроме Свободного чата) либо ПЕРЕДАЁТ, либо ПОЛУЧАЕТ хотя бы одну переменную. Если режим изолирован — это red flag
- [ ] **D5:** Переменные из ПЕРЕДАЁТ одного режима реально используются в промпте ПОЛУЧАЕТ другого. Проверка: найти имя переменной (напр. `life_position`, `user_game_formula`) в тексте промпта-получателя — оно должно быть в секции КРОСС-РЕЖИМНАЯ СВЯЗКА или ЛОГИКА ДИАЛОГА
- [ ] **D6:** Сводная таблица связок (из MODE_DETAILS) совпадает с тем, что реально прописано в промптах. Если в таблице написано «Разбор игры → Выход из игры: user_game_formula», то промпт «Выход из игры» должен содержать инструкцию использовать `user_game_formula`

## E. Интеграция с тестом (если у программы есть тест)

- [ ] **E1:** Результаты теста (scores_by_scale) инжектятся в контекст чата (`prepare-context.ts`)
- [ ] **E2:** Маппинг «шкала → режим» задокументирован (какой режим для какой шкалы primary)
- [ ] **E3:** В промптах режимов есть инструкция: если есть ISSP/тест данные → использовать для персонализации
- [ ] **E4:** AI не зачитывает баллы вслух — использует как навигацию (в промпте есть это правило)

## F. Интеграция с темами (если у программы есть program_themes)

- [ ] **F1:** `program_themes` имеют поле `recommended_route` (не null для всех тем)
- [ ] **F2:** Каждая тема ведёт в конкретный режим (не только в `/chat/new?topic=`)
- [ ] **F3:** Маппинг тема → режим логичен (тема «границы» → режим «тренировка границ», не «теория»)

## G. Ретроспектива и документация

- [ ] **G1:** Файл `examples/{slug}.md` в book-to-modes существует и актуален
- [ ] **G2:** Содержит секции: что сработало, что изобретали, специфика книги, уроки
- [ ] **G3:** Файл `{SLUG}_SYSTEM_PROMPTS.md` существует (промпты как MD-файл для ревью)
- [ ] **G4:** Файл `{SLUG}_MODE_DETAILS.md` существует (детализация режимов)

## H. Качество контента промптов (глубокая проверка)

- [ ] **H1:** Персона AI соответствует типу книги (терапия = бережный наставник, бизнес = коуч)
- [ ] **H2:** Тон единообразен во всех режимах одной программы
- [ ] **H3:** Промпты не содержат хардкод на конкретного пользователя или ситуацию
- [ ] **H4:** Ролевые режимы имеют: setup (2-3 обмена), маркер обратной связи (💡), дебрифинг
- [ ] **H5:** Экзаменационные режимы имеют: адаптивную сложность, metacognitive calibration, фидбэк через концепции книги
- [ ] **H6:** Лекционные режимы используют сократовский метод (вопросы → размышление → дополнение), не монологи
