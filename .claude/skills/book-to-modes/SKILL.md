# Book-to-Modes: Проектирование режимов AI-тренажёра по книге

## Обзор

Этот скилл превращает книгу по психологии/саморазвитию в набор из 6-8 режимов AI-тренажёра с готовыми системными промптами для Gemini API.

**Вход:** Название книги (+ опционально PDF/текст книги). Если у автора несколько связанных книг — можно объединить.
**Выход:** MD-файл с детализацией режимов + MD-файл с готовыми промптами + landing_data JSON + таблица технического маппинга

## Перед началом работы

1. Прочитай `references/REFERENCE.md` — 53 принципа, 5 архетипов, промпт-шаблон, типология replies, антипаттерны, принципы лендинга (Л1-Л7)
2. Прочитай `references/PLATFORM_MAP.md` — как режимы ложатся на код платформы (таблицы БД, chat_type, SQL, landing_data JSON-схема)
3. Просмотри `examples/` — конспекты решений по прошлым книгам (что сработало, что нет)
4. Если пользователь приложил PDF/текст книги — прочитай его
5. Если книга не приложена — используй свои знания. Для малоизвестных книг предупреди, что нужен текст

## Чеклист: объединение книг

Перед началом работы проверь — нужно ли объединять несколько книг в одну программу.

**Когда объединять:**
- У автора 2+ книги, которые дополняют друг друга (теория + практика, диагностика + лечение)
- Одна книга без другой «неполна» — читатель застревает на полпути
- Книги образуют естественную прогрессию

**Примеры:**
- Берн: «Игры» (1964) + «Люди, которые играют в игры» (1972) → объединены в одну программу
- Готтман: «7 принципов» + «The Relationship Cure» → можно объединить
- Одиночная книга → не объединять

**Как объединять:**
1. Определи части программы как прогрессию: «Увидеть → Понять → Действовать» (или другую)
2. Распредели режимы по частям
3. Маппинг: какая книга даёт материал для какой части
4. В таблице режимов добавь колонку «Часть программы»

**Когда НЕ объединять:**
- Книги на одну тему, но разных авторов (Гловер + Берн — это разные программы)
- Книга самодостаточна (Гловер «Славные парни» — одна книга = одна программа)

## Процесс (5 этапов с паузами)

### Этап 1: Анализ книги

Изложи:

**Ключевые концепции** — 3-5 главных идей книги. Не пересказ, а «что читатель должен усвоить».

**Навыки/трансформации** — что книга обещает. Формулируй как навык: «распознать X → перестроить Y → получить Z».

**ЦА** — пол, возраст, контекст, уровень боли (низкий = «хочу лучше», средний = «есть проблема», высокий = «кризис»).

**Боли** — 4-6 штук от первого лица: «Я стараюсь, а он/она...», «Не могу понять почему...». Это будущие welcome replies.

**Если объединяешь книги** — укажи какие концепции из какой книги, и как они складываются в прогрессию.

### Этап 2: Маппинг на архетипы

Определи **тип книги** и примени паттерн размножения (см. REFERENCE.md секция 1).

Формула: 5 базовых архетипов + 1-2 подвида самого релевантного = **6-8 режимов** + обязательный **Свободный чат**.

Предложи таблицу:

| # | Название | Архетип | Блум | Часть программы | Суть (1 предложение) |
|---|----------|---------|------|-----------------|----------------------|

Колонка «Часть программы» — обязательна для объединённых книг, опциональна для одиночных.

Объясни: почему именно так, что НЕ включил и почему.

**Решение о тесте:** Определи, нужен ли формализованный тест для книги. Критерии — см. REFERENCE.md секция 12 «Когда книге нужен тест». Если да — укажи: «Тест: да, ~N вопросов, K шкал» + предложи названия шкал и их связь с режимами. Если нет — укажи «Тест: нет» и почему (недостаточно шкал, книга не диагностическая, и т.д.).

**⏸ СТОП. Дождись подтверждения пользователя.**

### Этап 3: Детализация каждого режима

Для КАЖДОГО режима:

**a) Формат:**
- Архетип (из 5)
- Персона AI: тон, характер, обращение. Определяй по ЦА книги (см. таблицу per-book параметров в REFERENCE.md)
- Какие концепции из книги покрывает

**b) Welcome-сообщение** (строго по шаблону):
```
1. ЭМОДЗИ + НАЗВАНИЕ
2. ХУК: 1-2 предложения — продаёт результат, НЕ описывает процесс
3. КАК ЭТО РАБОТАЕТ: описание формата (1-2 предложения)
4. ЧТО ВЫ ПОЛУЧИТЕ: 3 bullet points (конкретные результаты)
5. SUGGESTED REPLIES: 3-4 варианта (голос пользователя, от первого лица)
   + 1 безопасный exit: «Мне сложно сформулировать»
```

**c) Паттерн диалога** — 3-5 фаз:
- Что происходит на каждой фазе
- Примеры вопросов AI
- Suggested replies на каждой фазе (тип из типологии — см. REFERENCE.md)
- Где scaffolding fading (4→2→0 replies)
- Где affect labeling («опиши чувство одним словом»)
- Где retrieval practice («что помнишь из прошлого раза?»)
- Где prior knowledge activation («вспомни ситуацию из жизни, когда...»)

**d) Кросс-режимная связка** (формализованный формат):
```
ПЕРЕДАЁТ → режим_name: variable_name (тип)
ПОЛУЧАЕТ ← режим_name: variable_name (тип)
```

Используй стандартные имена переменных из REFERENCE.md секция 9.

В конце этапа 3 — **сводная таблица кросс-режимных связок** (из → в → данные).

**⏸ СТОП. Дождись подтверждения пользователя.**

### Этап 4: Системные промпты для Gemini API

Для КАЖДОГО режима — готовый промпт. Структура:

```
## РОЛЬ
[Кто ты, тон, обращение]

## КНИГА
[Ключевые концепции для ЭТОГО режима — не вся книга, а релевантный срез]

## ПРАВИЛА ОТВЕТА
[Вставить промпт-шаблон из REFERENCE.md — секция «Промпт-шаблон»]

## WELCOME
[Welcome-сообщение из этапа 3b]

## ЛОГИКА ДИАЛОГА
[Фазы из этапа 3c: что делать, когда переходить, примеры вопросов]

## SUGGESTED REPLIES
[Конкретные replies для каждой фазы: текст + тип]

## НАУЧНЫЕ ПРИНЦИПЫ
- Process-level feedback (не «молодец», а «ты заметил X — точно, потому что Y»)
- Prior knowledge activation (перед темой: «вспомни ситуацию из жизни»)
- Affect labeling (перед глубокой темой: «опиши чувство одним словом»)
- Scaffolding fading (turns 1-5: 3-4 replies, 6-12: 2, 13+: 0-1)
- Desirable difficulties (при ошибке: наводящий вопрос, не ответ)

## КРОСС-РЕЖИМНАЯ СВЯЗКА
[Что передаёт / получает — формализованный формат]

## АНТИПАТТЕРНЫ (НИКОГДА)
- Стены текста (макс 60-80 слов)
- Похвала личности («молодец», «отлично»)
- Replies = дубликат текста AI
- Replies от третьего лица / вопросы AI
- Overdiagnosis (презумпция нормы)
- Геймификация (стрики, бейджи)
- Давление («ты пропустил», «давай быстрее»)
- Несколько вопросов за раз
- Предположения о поле/ориентации до уточнения
```

В конце этапа 4 — **таблица технического маппинга** (см. PLATFORM_MAP.md):

| Режим | mode_template.key | chat_type | route_suffix | icon | icon_exists? | sort_order |
|-------|-------------------|-----------|-------------|------|-------------|------------|

Колонка `icon_exists?` — проверь по списку в PLATFORM_MAP.md секция «Иконки режимов». Если ❌ → добавь задачу: создать иконку в **трёх файлах** (mode-icons.tsx, hub-icons.tsx, InstrumentList.tsx) + добавить toolKeyMap для chat-based режимов. Подробности в PLATFORM_MAP.md → «Чеклист при добавлении иконки».

**Если у программы есть Темы** (program_themes) — добавь таблицу маппинга тем → режимы (см. PLATFORM_MAP.md секция «program_themes → режимы»).

Каждый промпт должен быть **готовым для вставки** в Gemini API — не абстрактным, а рабочим. Оборачивай в блоки ``` для копирования.

### Этап 4.1: Создание теста (если решили «да» на этапе 2)

Если на этапе 2 решили, что тесту быть — создай полный `test_config` для вставки в БД. Формат и SQL-шаблон — см. PLATFORM_MAP.md секция «test_configs». Принципы дизайна вопросов — см. REFERENCE.md секция 12.

Для каждой шкалы:
- **Ключ** (snake_case, с префиксом книги) и **название** (русское)
- **5-7 вопросов** на шкалу (баланс direct/reverse)
- **Связь с режимами** — какой режим раскрывает эту шкалу (→ `exercises` в шкале)
- **radar_label** — 1-2 строки для отображения на радар-диаграмме

Дополнительно:
- **scoring** — `answer_range`, `score_direction`, `level_thresholds`, `level_labels`
- **ui_config** — `questions_per_block`, `auth_wall_question` (null если не нужна), `welcome_stats`, `block_insights`, `quick_answer_labels`, `analyzing_stages`
- **interpretation_prompt** — промпт для AI-интерпретации результатов (Gemini Pro). Формат: роль + книга + правила + JSON-схема ответа
- **mini_analysis_prompt_template** — шаблон для мини-анализа отдельного ответа (с плейсхолдером `{{questionText}}`)

Выход: готовый SQL INSERT в `test_configs` (оборачивай в ``` для копирования).

### Этап 4.5: Лендинг

На основе анализа из этапа 1 (ЦА, боли, концепции) и режимов из этапов 2-4. Принципы — см. REFERENCE.md секция 11 (Л1-Л7). Схема JSON — см. PLATFORM_MAP.md секция «programs.landing_data».

Для КАЖДОЙ секции:

**a) Hero** — главный оффер:
- `hero_title`: результат, не инструмент. Поддерживает `<em>` для акцента. Макс ~50 символов
- `hero_subtitle`: что конкретно делает тренажёр (1-2 предложения)
- `hero_cta`: текст CTA-кнопки (обычно «Начать бесплатно ↓» или «Узнать свой X ↓»)
- `hero_hint`: «Бесплатно, без регистрации» или «Без регистрации»
- `hero_tag`: обычно «AI-тренажёр по книге»

**b) Book** — обложка и автор:
- `cover_url`: найти книгу на **litres.ru** → ПКМ на обложку → «Копировать адрес изображения». Формат: `https://cdn.litres.ru/pub/c/cover_415/{id}.webp`
- `alt`, `author_top` (рус), `title` (рус), `subtitle` (оригинал), `author_bottom` (eng + credentials)

**c) Social proof** — 4 карточки `{icon, main, sub}`:
- Цифры методологии/книги, не платформы (Л4)
- Обязательно: тираж/охват + возраст методологии + «24/7» + ещё одно

**d) Problem + Personas** — боли ЦА:
- `problem.pain_cards`: 4 штуки, от первого лица (Л2). Из болей этапа 1
- `personas.items`: 4 штуки `{headline, body}` — сегменты ЦА

**e) Outcomes** — 5-6 результатов:
- Привязаны к конкретным режимам (Л6)
- Обязательные: ключевой результат книги + «Российский контекст» + «Всегда доступен» + «Дешевле одной сессии»
- Если `features.author_chat=true` → «Разговор с автором»

**f) Comparison** — таблица сравнения:
- 3 колонки: Книга, Профессионал (адаптировать под тему), AI-тренажёр (highlight)
- 5 строк: ключевые критерии + стоимость
- `conclusion`: позиционирование AI как «моста» (Л3)

**g) How it works** — 4 шага:
- `steps[].type`: `"chat"` | `"exercise"` | `"insight"` | `"portrait"`
- Адаптировать под features программы

**h) Author** (если `features.author_chat=true`):
- `photo_url`: найти фото автора в интернете. Приоритет: Wikipedia Commons → офф. сайт → издательство. Прямая ссылка на изображение
- `name`, `credentials` (1-2 предложения), `quote` (характерная цитата автора)

**i) Anonymous chat** (вне landing_data, но часть ленда):
- `anonymous_system_prompt`: короткий промпт для демо-чата (макс 3 обмена до auth wall). Зацепить интерес, не давать стену текста
- `anonymous_quick_replies`: 3-4 стартовых кнопки (голос ЦА)
- `chat_header`: `{title, subtitle}` — CTA для секции чата

**j) SEO и цены:
- `meta_title`: «AI-тренажёр: Название — Автор | Nice Guy AI»
- `meta_description`: 1-2 предложения
- `price`: `{trial_text, price_text, anchor_text}`

**k) Test** (только если `features.test=true`):
- `test`: `{emoji, title, description, time_label, questions_label, cta_text, cta_href}`

⏸ СТОП не нужен — ленд создаётся на основе утверждённых этапов 1-4.

### Этап 5: Ретроспектива

После завершения этапов 1-4 — запиши конспект решений в `examples/{book_slug}.md`:

1. **Что сработало** — удачные решения, которые стоит повторить
2. **Что пришлось изобретать** — дыры в скилле, которые нужно закрыть
3. **Что заняло больше всего времени** — кандидаты на шаблонизацию
4. **Специфика книги** — что уникально для этой книги и не обобщается
5. **Решения по объединению** (если книг несколько) — что было сложно, как решили

Этот конспект станет контекстом для следующих книг.

## Выходные файлы

Создай файлы и сохрани в проекте:

1. `{BOOK_SLUG}_MODE_DETAILS.md` — этап 3 (детализация: welcome, фазы, replies, связки)
2. `{BOOK_SLUG}_SYSTEM_PROMPTS.md` — этап 4 (готовые промпты для Gemini API)
3. `examples/{book_slug}.md` — этап 5 (конспект решений)

Где `{BOOK_SLUG}` — английское название книги в snake_case (напр. `5_love_languages`, `games_people_play`).
