# Runbook: Форматирование сообщений чата и quick replies

**Когда использовать:** При написании welcome-сообщений любого режима, правил в system_prompt, при аудите книги. Обязательно читать перед seed-SQL новой книги и при диагностике «почему кнопки не появляются».
**Время выполнения:** 5 минут чтения, применяется постоянно.
**Связанные:** [[add-new-book]], [[prompt-tuning]], [[../adr/004-context-assembly|ADR-004]].
**Source of truth:** этот файл. Скилл `book-to-modes` и `book-audit` ссылаются сюда.

## Контекст

На платформе **две разные поверхности рендеринга** AI-сообщений — у них разный пайплайн и разные требования к исходному тексту. Большинство багов возникает из-за того, что контент писался под одну поверхность, а попал на другую.

| Поверхность | Компонент | Markdown? | Где видно |
|-------------|-----------|-----------|-----------|
| **Welcome-экран нового чата** | [NewChatScreen.tsx](../../components/chat/NewChatScreen.tsx) | ❌ plain text | `/program/<slug>/chat/new?tool=...` — когда кликнул на инструмент на хабе |
| **Диалог в чате** | [ChatWindow.tsx](../../components/ChatWindow.tsx) | ✅ ReactMarkdown | `/program/<slug>/chat/<id>` — все последующие сообщения |
| **Первое сообщение в истории (legacy)** | ChatWindow через `welcomeMessage` prop | ✅ ReactMarkdown | Старые режимы, где welcome хранится в `welcome_message` |

## Архитектурный инвариант (🚨 ЧИТАТЬ ДО ЛЮБОЙ ПРАВКИ ЧАТОВ)

**Правило:** ВСЕ чат-поверхности (ChatWindow, NewChatScreen, AnonymousChat + любые новые) рендерят AI-сообщения через два компонента из [components/chat/ChatMessage.tsx](../../components/chat/ChatMessage.tsx):
- `<AIBubble>` — пузырь AI с ReactMarkdown + `remark-breaks`.
- `<QuickReplyBar>` — блок кнопок-«ёлочек».

Парсер «ёлочек» — единый: [lib/chat/parse-quick-replies.ts](../../lib/chat/parse-quick-replies.ts). Вызывающий код парсит один раз и передаёт `cleanText` → `<AIBubble>`, `replies` → `<QuickReplyBar>`.

**🔥 КРИТИЧНОЕ ПРАВИЛО LAYOUT:** `<QuickReplyBar>` обязательно рендерится как **SIBLING** контейнера сообщения (`.nc-msg` / `.msg`), а **НЕ ВНУТРИ** него. Контейнеры сообщений — flex-row (аватар слева, bubble справа); если положить кнопки внутрь, они станут третьим flex-элементом и уедут в узкую колонку справа. Используй `<Fragment>` чтобы вынести QuickReplyBar за пределы `.nc-msg` / `.msg`.

**Почему эти правила критичны:**
- До 2026-04-23 парсер «ёлочек» был локальным в ChatWindow. NewChatScreen и AnonymousChat рендерили AI-ответы через голый `<ReactMarkdown>` — кнопки-«ёлочки» НЕ появлялись до ручного reload (см. [docs/chat-audit-eq-2-0.md](../chat-audit-eq-2-0.md)).
- Промпты учат AI ставить «ёлочки» через одиночный `\n`. CommonMark трактует одиночный `\n` как пробел — «ёлочки» склеивались в одну строку. `remark-breaks` превращает одиночный `\n` в `<br>`.
- 2026-04-24: первая итерация единого компонента (`<ChatMessage>` монолит) рендерила bubble и кнопки в одном Fragment — кнопки ушли в узкую колонку сбоку. Разделение на `<AIBubble>` + `<QuickReplyBar>` с размещением как sibling — фикс этого.

**Запрещено:**
- ❌ Рендерить AI-ответ через `<ReactMarkdown>` напрямую в чат-компоненте. Всегда через `<AIBubble>`.
- ❌ Копировать логику `parseQuickReplies` в компонент. Импорт из `@/lib/chat/parse-quick-replies`.
- ❌ Ставить `<QuickReplyBar>` ВНУТРЬ `.nc-msg` / `.msg`. Только как sibling после закрывающего `</div>` контейнера сообщения.
- ❌ Создавать новый чат-экран без `<AIBubble>` + `<QuickReplyBar>`.

**Шаблон для нового чат-экрана:**
```tsx
import { Fragment } from "react";
import { AIBubble, QuickReplyBar } from "@/components/chat/ChatMessage";
import { parseQuickReplies } from "@/lib/chat/parse-quick-replies";

{messages.map((msg, idx) => {
  if (msg.role === "user") { /* обычный user bubble */ }

  const isLast = idx === messages.length - 1;
  const { cleanText, replies } = parseQuickReplies(msg.text, isLast && isStreaming);

  return (
    <Fragment key={msg.id}>
      <div className="msg msg-ai">          {/* flex-row: аватар + bubble */}
        <div className="msg-avatar ai" />
        <AIBubble text={cleanText} className="msg-bubble" />
      </div>
      {isLast && !isStreaming && (         {/* SIBLING, не внутри .msg-ai */}
        <QuickReplyBar
          replies={replies}
          onClick={handleSend}
          classNames={{ container: "quick-replies", button: "quick-reply-btn" }}
        />
      )}
    </Fragment>
  );
})}
```

**Нужна кастомизация которой нет в API?** Расширь API этих двух компонентов, не создавай обход. Любой обход = возврат к классу багов 2026-04-23.

## Три уровня контента

| Уровень | Поле БД | Для кого работает |
|---------|---------|-------------------|
| **Программный** | `programs.system_prompt`, `author_chat_system_prompt`, `free_chat_welcome`, `author_chat_welcome` | free_chat, author_chat, все **темы** (`program_themes`) — базовый system_prompt + welcome |
| **Режим** | `program_modes.system_prompt`, `welcome_ai_message`, `welcome_replies`, `welcome_message` | Tool-режимы (notes_*, ng_*, ta_* и т.п.) — переопределяет программный system_prompt |
| **Тема** | `program_themes.welcome_ai_message`, `welcome_replies`, `welcome_system_context` | Темы на хабе. welcome_system_context подмешивается к программному system_prompt через `"КОНТЕКСТ ТЕМЫ:"` префикс — см. [lib/chat/prepare-context.ts](../../lib/chat/prepare-context.ts) |

**Критично:** правила quick replies (блок «ФОРМАТ QUICK REPLIES» с «ёлочками», scaffolding, ЗАПРЕЩЕНО) нужно иметь **на каждом уровне, где AI генерирует ответы**:
- `programs.system_prompt` — покрывает free_chat и все темы.
- `programs.author_chat_system_prompt` — покрывает author_chat.
- `program_modes.system_prompt` — каждый tool-режим (переопределяет программный, поэтому правила не наследуются автоматически).

Если правила есть только в режимах, темы и свободный чат будут выдавать AI-ответы без «ёлочек».

## Как работают quick replies (критично)

### Welcome replies (кнопки на NewChatScreen)

Хранятся в `program_modes.welcome_replies` как JSONB-массив **объектов**:

```json
[
  {"text": "Я боюсь запустить проект — вдруг не получится", "type": "normal"},
  {"text": "Я знаю что нужно делать, но постоянно откладываю", "type": "normal"},
  {"text": "Мне сложно сформулировать свой страх", "type": "exit"}
]
```

- `type: "normal"` — обычная кнопка (golden).
- `type: "exit"` — safe exit, визуально мягче.
- Массив строк `["Текст"]` — тоже работает (legacy normalizer), но **не используй** — будет `type=normal` без контроля.

### In-chat replies (кнопки под ответами AI в диалоге)

**Единственный формат, который парсится:** «кавычки-ёлочки» (или прямые `"..."`) на отдельных строках в самом конце сообщения AI. Регэксп: `/^[«"](.+?)[»"]$/` — см. [ChatWindow.tsx:50-73](../../components/ChatWindow.tsx:50).

```
Основной ответ AI на 2 абзаца до 80 слов.

«Первый вариант от первого лица»
«Второй вариант»
«Мне сложно сформулировать»
```

Парсер:
1. Читает строки с конца, пропускает пустые.
2. Собирает те, что матчат `«...»` или `"..."`.
3. Как только встречает не-матч — останавливается.
4. Собранные строки вырезает из `cleanText` и отображает как кнопки.

**Что НЕ работает (рендерится как markdown-текст, кнопок не будет):**

- ❌ Блок «Что дальше?» с вопросами списком.
- ❌ Варианты через `—` em-dash в начале строки.
- ❌ Markdown-списки через `-` / `*` / `•`.
- ❌ Нумерованные списки `1. ... 2. ...`.
- ❌ Reply внутри параграфа, не на отдельной строке.
- ❌ Reply в середине сообщения (парсер берёт только с конца).
- ❌ **Склеенные «ёлочки» через пробел на одной строке** — частый баг: модель иногда выдаёт `«Текст 1» «Текст 2» «Текст 3»` в одной строке. Каждая должна быть на отдельной.
- ❌ «Ёлочка» с trailing-пунктуацией снаружи: `«Текст».` — точка вне кавычек ломает regex.
- ❌ Вложенные «ёлочки»: `«Что для тебя значит «развитие через удовольствие»?»` — non-greedy regex захватит только до первого внутреннего `»`. Внутренние кавычки нужно убирать.

## Правила по полям

### `welcome_mode_label` — плашка над заголовком

- Одно слово UPPERCASE: `АНАЛИЗ`, `ЛЕКЦИЯ`, `ВОРКШОП`, `РОЛЕВАЯ`, `ЭКЗАМЕН`, `СВОБОДНЫЙ ЧАТ`, `РАБОТА С ТЕМОЙ`.
- Соответствует архетипу из `book-to-modes` → REFERENCE.md §1.

### `welcome_title` — заголовок карточки

- Название режима. **Без эмодзи.** Эмодзи уже могут быть на иконке инструмента на хабе, в заголовке — только текст.
- Пример: `Деконструктор страхов`, `Масштаб мышления`, `Архитектура энергии`.

### `welcome_subtitle` — подзаголовок карточки

- Одна строка (≤80 символов). Результат-обещание, не описание процесса.
- ✅ «Расшатай финансовый потолок через серию провокационных вопросов».
- ❌ «Мы разберём твой страх».

### `welcome_ai_message` — основное сообщение на welcome-экране ⚠️ PLAIN TEXT

Самая частая точка багов. Рендерится **БЕЗ** ReactMarkdown — [NewChatScreen.tsx:185](../../components/chat/NewChatScreen.tsx:185):

```tsx
<div className="nc-ai-text">{welcome.aiMessage}</div>
```

**Правила:**

| Правило | Комментарий |
|---------|-------------|
| Никакого `**bold**` / `*italic*` / `#` / `-` | Звёздочки видны буквально в UI |
| Эмодзи в тексте — **ок**, но **не в самом начале с заголовком** | `🔒 **Деконструктор страхов**` — двойной баг: markdown + дублирование title |
| Не дублировать `welcome_title` | Заголовок уже показан в W2 над ai_message |
| Переносы через `\n\n` между абзацами | Иначе всё склеивается в один блок |
| Простой список через `•` (буллет-символ, не markdown `-`) | Буллет рендерится как есть |
| Макс ~150 слов суммарно | Длиннее = стена |

**Канонический шаблон:**

```
[Хук-результат на 1-2 предложения — продаёт что получит пользователь]

Как это работает: [1-2 предложения про формат]

Что ты получишь:
• [конкретный результат 1]
• [конкретный результат 2]
• [конкретный результат 3]
```

**Антипример (как сейчас в 100-notes):**

```
🔒 **Деконструктор страхов** Страх — не твой враг. Это компас... **Что ты получишь:** • Чёткое понимание...
```

Рендерится так: `🔒 **Деконструктор страхов**` (буквально со звёздочками), всё на одной строке, без переносов.

**Правильно:**

```
Страх — не твой враг. Это компас, который точно указывает на зону роста.

Мы разберём один конкретный страх, который сейчас тебя останавливает, и увидим, что он на самом деле скрывает.

Что ты получишь:
• Чёткое понимание, какую зону роста подсвечивает твой страх
• Разоблачение вторичных выгод — почему тебе «выгодно» бояться
• Конкретный первый шаг, который можно сделать прямо сейчас
```

### `welcome_replies` — кнопки-подсказки на welcome

См. выше «Welcome replies». Ключевое: **объекты `{text, type}`**, не строки. Последний — `type: "exit"` = «Мне сложно сформулировать» или аналог.

### `welcome_message` — legacy, первое сообщение в истории чата

- Рендерится через ReactMarkdown — markdown поддерживается.
- Если нужны стартовые кнопки в чате — добавь `«ёлочки»` на отдельных строках в конце.
- Для новых режимов предпочитай структурированный путь (`welcome_ai_message` + `welcome_replies`), `welcome_message` оставляй только для persist в истории.

### `programs.free_chat_welcome` / `author_chat_welcome` — стартовые «ёлочки»

Эти поля показываются как первое AI-сообщение в ChatWindow для **свободного чата** и **разговора с автором**. Чтобы были стартовые кнопки — в конец welcome-текста добавь «ёлочки» на отдельных строках:

```
Привет! [текст welcome].

С чего начнём?

«Стартовый вопрос 1?»
«Стартовый вопрос 2?»
«Стартовый вопрос 3?»
«Просто хочу поговорить»
```

Без «ёлочек» в тексте `parseQuickReplies` не найдёт кнопок и пользователь увидит только голое сообщение.

### `program_themes` — темы на хабе

Тема переиспользует `programs.system_prompt` + подмешивает свой `welcome_system_context`. Поэтому:
- Правила quick replies должны быть в `programs.system_prompt` (не в `welcome_system_context` каждой темы).
- `welcome_ai_message` темы — отображается на NewChatScreen как plain text (те же правила что и для режима: без `**`, без дублирования title).
- `welcome_replies` темы — формат `[{text, type}]`.

### `system_prompt` режима — правила, которые удержат AI от нарушений M1–M10

Каждый system_prompt должен содержать **явный блок о формате quick replies** с буквальным примером. Без него модель сползает в «Что дальше?» / «—»-списки / нумерацию.

**Обязательный блок для каждого промпта:**

````
## ФОРМАТ ОТВЕТА

### Длина
- Максимум 2 абзаца, 60–80 слов.
- Один вопрос за ход. Никогда не задавай 2–3 вопроса в одном сообщении.

### Quick replies
Если хочешь предложить пользователю варианты ответа — пиши их
В КОНЦЕ сообщения, каждый на отдельной строке, в ёлочках:

«Вариант 1 от первого лица»
«Вариант 2»
«Мне сложно сформулировать»

Это ЕДИНСТВЕННЫЙ формат, который рендерится как кнопки.

### ЗАПРЕЩЕНО
- Блок «Что дальше?» со списком вопросов.
- Список вариантов через «—» или «-» — это не станет кнопками.
- Нумерованные списки «1. ... 2. ...» как варианты.
- Несколько вопросов за ход.
- Похвала личности («молодец», «умница»). Только похвала стратегии.

### Scaffolding fading
- Turns 1–5: 3–4 reply.
- Turns 6–12: 2 reply.
- Turns 13+: 0 reply или 1 fallback exit.
````

## Диагностика живых багов

**«Кнопок не появилось в welcome-экране»**
→ `welcome_replies` — строки вместо объектов? Пустой массив? Проверь SQL: `SELECT welcome_replies FROM program_modes WHERE ...`.

**«В welcome-экране видны `**`»**
→ `welcome_ai_message` содержит markdown. Плейн-текст-рендер — убрать `**`.

**«В welcome-экране заголовок повторяется два раза»**
→ `welcome_ai_message` начинается с эмодзи + названия режима. Удалить этот префикс.

**«AI в диалоге выдаёт список вариантов, но кнопок нет»**
→ AI использует «Что дальше?» / `—` / `-` / нумерацию вместо «ёлочек». Починка — в `system_prompt`: добавить блок `## ФОРМАТ ОТВЕТА` с буквальным примером.

**«AI выдаёт «ёлочки», но они всё равно отображаются как текст»**
→ Модель склеила их в одну строку через пробел: `«Вариант 1» «Вариант 2»`. Регэксп требует каждую на отдельной строке. Починка — в `system_prompt` добавить явное правило: «Перед первой «ёлочкой» — пустая строка, между «ёлочками» — одиночный перенос. НИКОГДА не склеивай через пробел», плюс контрпример «НЕПРАВИЛЬНО».

**«В свободном чате / разговоре с автором нет кнопок»**
→ `programs.free_chat_welcome` / `author_chat_welcome` не содержат «ёлочек». Добавить в конец welcome-текста.

**«В теме AI выдаёт варианты списком без «ёлочек»»**
→ `programs.system_prompt` не содержит правил quick replies. Темы полагаются на программный промпт — добавить блок «ФОРМАТ QUICK REPLIES» туда.

**«После перехода из NewChatScreen в ChatWindow показывается generic «Привет! Я — AI-ассистент...» вместо mode-specific welcome»**
→ Баг в [app/program/[slug]/(app)/chat/[chatId]/page.tsx](../../app/program/[slug]/(app)/chat/[chatId]/page.tsx): если код не делает lookup по `chat.chat_type` в `program_modes`, welcome берётся из `programs.free_chat_welcome`. Проверь что страница ищет mode-specific welcome через JOIN с `mode_templates.key = chat_type` для tool-режимов.

**«AI задаёт по 3 вопроса за ход»**
→ Отсутствует правило `### Длина → Один вопрос за ход` в `system_prompt`.

**«Ёлочки» на welcome / в AI-ответе склеиваются в одну строку через пробел»**
→ Компонент рендерит AI-текст через ReactMarkdown **без `remark-breaks`**. CommonMark трактует одиночный `\n` как пробел. Проверь что импорт стоит: `import remarkBreaks from "remark-breaks";` + `<ReactMarkdown remarkPlugins={[remarkBreaks]}>`. Правильнее — использовать `<ChatMessage>` вместо прямого ReactMarkdown, он уже с `remark-breaks`.

**«Кнопки-«ёлочки» появляются только после reload страницы»**
→ Компонент рендерит AI-ответы через `<ReactMarkdown>` напрямую, а не через `<ChatMessage>`. После reload открывается ChatWindow (где парсер встроен), до reload — NewChatScreen/AnonymousChat без парсера. Переведи компонент на `<ChatMessage>` (см. архитектурный инвариант выше).

**«AI использует `<угловые скобки>` вместо «ёлочек»»**
→ В `system_prompt` этого уровня слабый QR-блок (нет контрпримера «НЕПРАВИЛЬНО: `<текст>`»). Частая причина — plейсхолдер `«Вариант 1 от первого лица»` в позитивном примере без тематической замены, Gemini ищет альтернативу. Фикс: заменить плейсхолдер на тематические примеры + добавить контрпример `НЕПРАВИЛЬНО: <вариант>`.

**«AI использует дефис-список `- "текст"` вместо «ёлочек»»**
→ Аналогично: слабый QR-блок без запрета на дефис-списки и прямые кавычки. ReactMarkdown видит `-` в начале строки и рендерит как `<ul><li>`. Фикс: добавить `НЕПРАВИЛЬНО: - "текст"` и `НЕПРАВИЛЬНО: "текст"` (прямые кавычки) в секцию ЗАПРЕЩЕНО.

**«AI оборачивает «ёлочки» в буллет-список `* «текст»`»**
→ Парсер с 2026-04-24 **толерантен**: срезает ведущие `* `, `- `, `• `, `1. ` перед match-ом regex. Кнопки появятся, даже если AI добавил buллет. НО: markdown-обёртка — признак слабого QR-блока в system_prompt. Всё равно усилить промпт (явный пример ПРАВИЛЬНО без маркеров + запрет `* «...»` в НЕПРАВИЛЬНО), чтобы модель не плодила мусор. Толерантность парсера — защита от случайных сбоев, не оправдание слабого промпта.

**«AI буквально пишет «Вариант 1»/«Вариант 2» в ответах»**
→ В примере QR-блока остались literal-плейсхолдеры. По правилу из скилла `book-to-modes` §5, плейсхолдеры `«Вариант 1»`, `«Вариант 2»` в ПРАВИЛЬНО-примере должны быть заменены тематически-конкретными reply для режима. Они могут остаться ТОЛЬКО в контрпримере `НЕПРАВИЛЬНО` (там наоборот — пусть модель видит их как «плохой формат»).
